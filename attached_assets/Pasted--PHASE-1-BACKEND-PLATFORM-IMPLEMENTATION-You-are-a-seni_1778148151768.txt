# PHASE 1 BACKEND PLATFORM IMPLEMENTATION

You are a senior backend platform engineer tasked with building Phase 1 of a reusable backend infrastructure platform using Django and Django REST Framework.

This project is NOT a simple CRUD Django application.

It is:

* a backend platform
* a reusable infrastructure layer
* an internal developer platform
* an API-first system
* a foundation for future products and services

The goal is to create a modular backend platform that exposes reusable APIs for services commonly required by applications, such as:

* Authentication
* Organizations/Tenants
* API Keys
* File Storage
* Email Services
* Audit Logging

Applications should consume these services through APIs instead of rebuilding them repeatedly.

The implementation must prioritize:

* maintainability
* modularity
* scalability
* extensibility
* clean architecture
* portability
* developer experience

The platform will initially run on constrained/low-cost environments such as PythonAnywhere Free Plan, where Redis and Celery may not be available.

The architecture MUST therefore:

* support lightweight execution initially
* remain production-ready architecturally
* support future migration to Celery/Redis without major rewrites

---

# PRIMARY OBJECTIVES

The completed Phase 1 platform must allow developers to:

1. Register and authenticate users
2. Create organizations/workspaces
3. Generate and manage API keys
4. Upload and manage files
5. Send transactional emails
6. Access APIs securely
7. Track audit logs

# ABSOLUTE ARCHITECTURAL RULES

## RULE 1 — USE MODULAR MONOLITH ARCHITECTURE

The system MUST be implemented as a modular monolith.

DO NOT implement microservices.

DO NOT split into separate repositories.

DO NOT introduce distributed systems architecture.

All apps must live inside one Django project while remaining logically separated.

---

## RULE 2 — BUSINESS LOGIC MUST NEVER LIVE IN VIEWS

Views/controllers must remain thin.

Views should:

* validate requests
* call services
* return responses

Business logic belongs ONLY in service layers.

BAD:

```python
class UploadView(APIView):
    def post(self, request):
        # large business logic here
```

GOOD:

```python
class UploadView(APIView):
    def post(self, request):
        StorageService.upload(...)
```

---

## RULE 3 — CREATE A SERVICE LAYER

Every major app MUST include:

* services.py
* tasks.py
* api/
* models.py

Optional:

* selectors.py
* repositories.py

Services contain business logic.

---

## RULE 4 — DO NOT COUPLE CODE TO DJANGO-Q2

The platform MUST NOT directly scatter:

```python
async_task(...)
```

throughout the codebase.

Instead, create an internal abstraction layer:

```python
TaskDispatcher.dispatch(...)
```

The entire platform must interact ONLY with the dispatcher abstraction.

This allows future migration to:

* Celery
* Redis
* AWS SQS
* Kafka
* Dramatiq

without rewriting business logic.

---

## RULE 5 — ALL APIs MUST BE VERSIONED

Every API route MUST begin with:

```txt
/api/v1/
```

Never create unversioned APIs.

---

## RULE 6 — USE CUSTOM USER MODEL IMMEDIATELY

The platform MUST implement a custom user model from the start.

DO NOT use Django’s default User model directly.

The user model must support future extensibility for:

* email login
* phone login
* OAuth
* MFA
* SSO
* account verification

---

## RULE 7 — ORGANIZATION/TENANT-FIRST DESIGN

Every core resource must belong to an organization/workspace.

Examples:

* API keys
* files
* emails
* logs

must be organization-aware.

Use shared database multi-tenancy with organization foreign keys.

DO NOT implement schema-based tenancy yet.

---

## RULE 8 — NEVER STORE RAW API KEYS

API keys must:

* be generated securely
* be hashed before storage
* never be retrievable again after creation

Similar to password handling.

---

## RULE 9 — USE PROVIDER ABSTRACTIONS

Email and storage systems MUST use provider architectures.

DO NOT tightly couple logic to:

* AWS
* Google Drive
* SendGrid
* local storage

Instead create provider interfaces.

Example:

```python
class BaseStorageProvider:
    def upload(...)
    def delete(...)
    def generate_signed_url(...)
```

---

## RULE 10 — ALL HEAVY TASKS MUST BE ASYNC-CAPABLE

Email sending, webhook delivery, and processing jobs must be dispatchable asynchronously through the internal task dispatcher.

Even if tasks execute synchronously during development, the architecture must remain async-ready.
